Perso XXL agent harness research

Build a testing environment first, not a bigger prompt.

The current extension proves the concept, but reliability will come from an eval loop: reproducible pages, controlled browser runs, clear oracles, trace capture, and a small holdout set that the optimizing agent cannot tamper with.

Executive Take

Use a local "web zoo" as the main benchmark: a small app with YouTube-like, Docs-like, dashboard, article, ecommerce, and weird dynamic DOM pages. Drive the Perso XXL extension through Playwright, collect before and after DOM, accessibility, screenshot, console, plan, and matched-node traces, then score each task with deterministic assertions first. Screenshots and visual graders are useful, but the core harness must also work with text-only models.

Best near-term objective: create a repeatable experiment loop where a harness change can be run against the same train/dev tasks in minutes, while a hidden holdout stays locked. This lets you learn whether a new tool, prompt, schema rule, selector strategy, or verifier actually improves the agent.

What The Demo Already Gets Right

  • Plans are declarative JSON, not arbitrary model-generated JavaScript.
  • The prompt asks the model to use targetMap selectors and typed rules instead of putting raw selectors directly on actions.
  • The picker snapshot includes hierarchy candidates, nearby text, computed style, selector hints, and a semantic-target guess.
  • The executor logs matched elements and can reapply plans after page mutations.
  • Validation blocks some unsafe CSS and invalid selectors before applying a plan.

Where Reliability Leaks

  • totalMatched > 0 is treated as progress, but matching anything is not the same as matching the intended element.
  • The DOM summary is capped at 220 visible nodes, so below-fold or late-rendered targets can disappear from context.
  • The agent gets text and DOM hints, but not a first-class accessibility tree, layout summary, or optional visual verification loop.
  • Plan repair fixes schema validation, not failed outcomes. There is no "observe result, diagnose, repair selector" cycle.
  • The current rule set is intentionally narrow. That is good for safety, but hard tasks like scroll control, collision, animation, or page behavior need trusted capability modules.

Bad Ideas To Avoid

Making live YouTube and Google Docs the core benchmark

They are useful smoke tests, but bad primary evals: unstable DOMs, accounts, localization, experiments, ads, and rate limits will make the metric noisy.

Using one VLM judge as the score

Vision judges are useful for subjective theming, but they are too easy to fool and too noisy for simple tasks. Prefer DOM and computed-style assertions where possible.

Rewarding "number of changes"

This directly incentivizes noisy page-wide CSS, broken layouts, and theatrical output. Score intent completion, blast radius, reversibility, and whether important controls still work.

Letting the optimizing agent edit the evals

That is the shortest path to fake improvement. Evals, expected labels, and judges need a write-protected boundary.

Giving arbitrary JavaScript as the default tool

It will solve demos and create security, persistence, and reward-hacking problems. Add trusted capability modules for behavior changes instead.

Defining easy, medium, hard by vibes

Keep the labels, but attach capability tags: selected target, inferred target, external lookup, ambiguity, dynamic DOM, visual design, behavior change, and multi-step repair.

The Web Zoo

Build a local app that simulates the product surfaces you care about without depending on live sites. Each page should have randomized variants so selector strategies are tested against realistic drift.

  • YouTube-like feed: cards, thumbnails, channel names, typo-prone creator names, lazy rendering, infinite scroll, hidden menus.
  • Docs-like home: template gallery, ugly nested roles, listboxes, generated class names, responsive layouts.
  • Dashboard: text-only panels that need semantic icons, SVG insertion, accessibility labels.
  • Theme page: many components where global design changes can succeed or break layout.
  • Physics page: controlled sandbox for collision/gravity behavior, with a trusted behavior module.
  • Adversarial page: decoy text, hidden labels, misleading classes, canary elements that must not change.

The Agent Interface Under Test

Do not test only the model response. Test the whole product path: prompt plus attachments or picked elements, DOM summarization, plan generation, validation, execution, reapply after mutation, and result verification.

  • Load Perso XXL as an unpacked Chrome extension in Playwright.
  • Inject prompts through a test bridge instead of relying on brittle manual UI clicks.
  • Simulate element picks by passing actual fixture DOM nodes to the extension picker path.
  • Capture extension logs from the local logger and browser console.
  • Record every generated plan and every selector match.

Task Set Design

Task pyramid

Hard Behavior changes, large themes, physics, subjective visual quality.
Medium Unselected targets, ambiguity, external lookup, semantic assets.

Tier Example Primary oracle What it tests
Easy Change this selected button text color to red. Computed style on hidden ground-truth target equals expected color; sibling controls unchanged. Selection grounding, selector generation, style application, blast radius.
Easy Delete this selected promo block. Ground-truth block hidden or removed; layout still has no overlap. Visibility rule correctness and layout safety.
Medium I do not want to see videos from "Mark Ruber" in my feed. Agent asks a clarification or resolves "Mark Rober" through approved lookup, then hides only that creator. Ambiguity handling, external lookup, confirmation before action.
Medium Add SVG icons to this dashboard. Expected panels receive semantically appropriate icons from approved icon set; accessible names exist. Asset/tool use, semantic design, non-destructive insertion.
Medium Remove the model gallery, I always start from a new document. Gallery region removed on Docs-like page; new document button remains visible and clickable. Unselected target inference on ugly DOM.
Hard Stop me from scrolling YouTube; show only the first loaded video. Scroll position cannot advance; only first feed item visible; page controls still usable. Trusted behavior capability, persistence, side effects.
Hard Give this page a One Piece theme. Hybrid score: DOM/theming coverage, layout invariants, screenshot diff, optional VLM rubric. Large visual transform without breaking page function.
Hard Apply gravity and collision to all elements. Uses approved physics module; elements animate in bounded container; controls remain accessible. Recognizing when schema needs a capability, not raw JS improvisation.

Scoring Model

Task success Did the requested change happen on the intended target?
Grounding Did selectors point to the semantic target, not a decorative child or unrelated shared class?
Blast radius How many non-target nodes changed? Did text, layout, or clickability break?
Persistence Does the change survive rerender, lazy loading, route changes, and viewport changes?
Interaction safety Can important controls still be clicked, typed into, and read by accessibility APIs?
Integrity Did the agent avoid modifying eval files, reading holdout labels, or exploiting grader artifacts?
Cost and latency Tokens, tool calls, wall time, browser actions, and retries.

Grader Stack

  1. Plan validator: schema, unsafe CSS, selector validity, capability permissions.
  2. DOM oracle: hidden server-side target map, expected element count, node visibility, attributes.
  3. Computed style oracle: exact colors, backgrounds, dimensions, overflow, pointer events.
  4. Accessibility oracle: role/name preservation, no hidden primary controls, tab order checks.
  5. Visual oracle: optional screenshot diff, overlap detection, viewport-specific checks.
  6. LLM/VLM judge: optional and only for subjective tasks, with a tight rubric.

Reward Hacking Controls

Keep this simple at first. A PROGRAM.md rule saying what the optimizer must not touch is enough for the first manual experiments. Add stronger controls only when we actually automate optimizer runs or see suspicious behavior.

Phase Build Exit criterion
0. Baseline Run 15 to 25 manual representative tasks against the current extension and save plans, logs, screenshots, and notes. Known failure taxonomy: wrong target, no target, broad selector, schema limit, visual break, persistence break.
1. Web zoo MVP Create local fixture app with 6 pages and 30 tasks split across easy, medium, hard. Every task has a deterministic reset, seed, target oracle, and browser URL.
2. Runner Playwright loads Chromium with Perso XXL, sends prompt/picks/assets, waits for completion, captures traces. One command runs the full public suite and writes JSON plus HTML report.
3. Graders Implement DOM, computed style, accessibility, and optional visual diff adapters. Easy tasks are deterministic; subjective hard tasks can add screenshots or VLM rubrics when available.
4. Harness experiments Try changes to context extraction, accessibility tree, layout summaries, optional screenshot labels, selector repair, and capability modules. Each experiment produces comparable task, cost, and latency metrics.
5. Autoresearch loop Let an agent propose harness changes against train/dev while following PROGRAM.md. Only keep patches that improve trusted metrics and pass manual diff review.

Tool Experiments To Try

  • Accessibility tree tool: expose role, accessible name, hierarchy, visibility, and clickability.
  • Target probe tool: given a selector, return count, sample text, bounds, styles, and whether it overlaps the user-selected target.
  • Layout summary tool: expose bounding boxes, stacking, overlap, and viewport position as text for non-visual models.
  • Screenshot crop tool: optional crop around selected or candidate nodes for visual-capable models.
  • Selector repair tool: after zero or wrong match, search by fingerprint and nearby text.
  • Capability tools: trusted modules for scroll locking, icon insertion, themes, and physics.
  • Clarification tool: require a question when intent is ambiguous or external lookup has low confidence.

What Not To Implement Yet

  • Do not build a full fake internet before the first benchmark. Six high-quality pages beat fifty shallow pages.
  • Do not optimize prompts before measuring the current prompt. Otherwise you cannot tell what helped.
  • Do not add broad JavaScript execution as a quick fix for hard tasks. Add named, auditable capabilities.
  • Do not make holdout huge at first. A small set is enough to keep us honest.
  • Do not rely on browser-use to verify your extension. Use it to compare agent-control approaches, not as the trusted judge.
